#+TITLE: Pitfalls and Drift Between Build Paths

* Fast rebuilds versus conventional Docker builds
The Nix path uses ~nix-build docker.nix~ and then ~docker load < result~.
That does /not/ use the ~Dockerfile~ at all.

The conventional path uses ~docker build~ against ~Dockerfile~.

If one file changes but the other does not, the two images can drift
silently.

* Current deliberate differences
These are differences we are currently tolerating on purpose.

** TypeDB installation method
The ~Dockerfile~ installs TypeDB from the vendor apt repository.

The Nix image builds TypeDB from ~typedb.nix~, then reconstructs a
writable ~\/opt/typedb~ tree at image-customisation time. It also
provides both of these paths:
- ~\/opt/typedb/server/data~
- ~\/opt/typedb/core/server/data~

That compatibility layer exists because scripts and docs in the repo
have historically referred to both layouts.

** Rust installation method
The ~Dockerfile~ installs Rust via the rustup shell installer and keeps
tooling under ~\/usr/local~.

The Nix image installs Rust tooling from nixpkgs and points
~RUSTUP_HOME~ and ~CARGO_HOME~ at ~\/home/ubuntu~.

** Claude/Codex installation method
Both build paths install Claude Code and Codex via ~npm install -g~,
but at different times and into different places.

The ~Dockerfile~ installs them at image-build time.

The Nix image does /not/ bake the CLIs in. Instead it ships the
wrapper scripts in ~copy-when-rebuilding/bin~ (~claude~, ~codex~,
~update-ai-clis~). On first use ~update-ai-clis~ runs
~npm install -g @anthropic-ai/claude-code@latest @openai/codex@latest~
into the writable prefix ~\/home/ubuntu/.local/npm-global~, and
~.bashrc~ re-runs it once a day. This keeps the CLIs on ~@latest~,
upgradable independently of nixpkgs.

Note: this is a deliberate departure from an earlier plan to use the
nixpkgs ~claude-code~ package plus a pinned Codex tarball. If I ever
switch back to nixpkgs-packaged CLIs, the FHS loader shim described
under "Operational pitfalls" will no longer be needed (nixpkgs patches
its binaries for the Nix store).

** Python and Node package shape
The ~Dockerfile~ explicitly installs ~python3-pip~, ~python3-venv~,
~pipx~, ~nodejs~, and ~npm~.

The Nix image installs ~python3~, ~pipx~, and ~nodejs_24~.
Those may be functionally fine while still not matching one-for-one.

** Base tool inventory
The Nix image explicitly includes several general-purpose tools not
spelled out in the ~Dockerfile~, including ~wget~, ~less~, ~procps~,
~psmisc~, ~gnused~, ~gnugrep~, ~findutils~, ~gawk~, ~which~,
~diffutils~, and ~cmake~.

The ~Dockerfile~ explicitly includes several bootstrap or apt-era
packages not mirrored one-for-one in ~docker.nix~, including
~default-jre~, ~build-essential~, ~libssl-dev~,
~apt-transport-https~, and ~software-properties-common~.

* Operational pitfalls
** when editing the Dockerfile
put new code as far at the end as possible,
unless there's lots of time to rebuild in a prettier order

** when editing docker.nix
If TypeDB won't start, check whether the image still provides a
writable ~\/opt/typedb~ tree and a writable data directory behind:
- ~\/opt/typedb/server/data~
- ~\/opt/typedb/core/server/data~

~bash/run-servers.sh~ depends on that runtime shape.

** the FHS dynamic loader shim is load-bearing
~docker.nix~ creates ~\/lib64/ld-linux-x86-64.so.2~ as a symlink to
nixpkgs glibc, and adds glibc, libstdc++, zlib and openssl to
~LD_LIBRARY_PATH~ (see ~aiCliLibraryPath~).

This exists because the npm-installed ~claude~/~codex~ are prebuilt FHS
binaries with the interpreter ~\/lib64/ld-linux-x86-64.so.2~ hardcoded.
A pure-Nix image has no such path, so without the shim ~execve()~ fails
with ENOENT, which the shell reports as
~cannot execute: required file not found~.

Do not remove the ~lib64~ symlink or the AI-CLI entries in
~LD_LIBRARY_PATH~ while the CLIs are installed from npm. Do not patch
the binaries in place instead: the ~claude~ binary is Bun-compiled with
data appended past the ELF image, and ~patchelf~ can corrupt it.
